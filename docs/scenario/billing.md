# Billing Scenarios (User-Facing)

このドキュメントは、課金まわりのユーザー視点シナリオを網羅します。  
期待値はUI/体験ベースで記載し、DB観点は含めません。

## S1: 初回課金 (Free -> Creator)
- 初期状態: 新規ユーザー。プランはFree。`/plans` を開いている。
- 行動: Creatorプランの「選ぶ」をクリック -> Stripe Checkoutで支払い完了。
- 期待値:
  - `/plans?checkout=success` で戻る。
  - 「処理中」オーバーレイが表示される。
  - しばらくして「現在のプラン」バッジがCreatorに付く。
  - 成功トーストが表示される。

## S2: 初回課金 (Free -> Business)
- 初期状態: 新規ユーザー。プランはFree。`/plans` を開いている。
- 行動: Businessプランの「選ぶ」をクリック -> Stripe Checkoutで支払い完了。
- 期待値:
  - `/plans?checkout=success` で戻る。
  - 「処理中」オーバーレイが表示される。
  - しばらくしてBusinessが「現在のプラン」になる。
  - 成功トーストが表示される。

## S3: 初回課金キャンセル
- 初期状態: Freeユーザー。`/plans` を開いている。
- 行動: 有料プランの「選ぶ」 -> Stripe Checkoutでキャンセル。
- 期待値:
  - `/plans?checkout=canceled` で戻る。
  - 「決済をキャンセルしました」トーストが表示される。
  - プラン表示はFreeのまま。

## S4: アップグレード (Creator -> Business)
- 初期状態: Creatorユーザー。`/plans` を開いている。
- 行動: Businessプランの「選ぶ」 -> Stripe Checkoutで支払い完了。
- 期待値:
  - `/plans?checkout=success` で戻る。
  - 「処理中」オーバーレイが表示される。
  - しばらくしてBusinessが「現在のプラン」になる。
  - 成功トーストが表示される。
  - Stripe側の請求が重複せず、日割り調整が反映される。

## S5: アップグレード中の順序入れ替わり耐性
- 初期状態: Creatorユーザー。アップグレード操作を実行中。
- 行動: Webhookの到着順が前後する状況が発生。
- 期待値:
  - 最終的にBusinessが「現在のプラン」として表示される。
  - Freeに落ちたままにならない。

## S6: ダウングレード (Business -> Creator, 上限内)
- 初期状態: Businessユーザー。所有ファンマ数がCreator上限以内。
- 行動: Creatorプランの「選ぶ」 -> 警告確認 -> 実行。
- 期待値:
  - 警告ダイアログの同意が必要。
  - 旧プランの権利が即時剥奪される旨がUIに表示される。
  - 処理後、Creatorが「現在のプラン」になる。
  - 成功トーストが表示される。

## S7: ダウングレード (Business -> Creator, 上限超過)
- 初期状態: Businessユーザー。所有ファンマ数がCreator上限を超過。
- 行動: Creator選択 -> 選択モーダルで残すファンマを選択 -> 返却確定。
- 期待値:
  - 選択モーダルが表示され、上限数まで選べる。
  - 未選択分は返却されることが明確に表示される。
  - 確定後にCreatorが「現在のプラン」になる。
  - 成功トーストが表示される。

## S8: ダウングレード (有料 -> Free)
- 初期状態: CreatorまたはBusinessユーザー。
- 行動: Freeへの変更 -> 警告確認 -> 実行。
- 期待値:
  - 旧プランの権利が即時剥奪される旨がUIに表示される。
  - Freeが「現在のプラン」になる。
  - 成功トーストが表示される。

## S9: Customer Portalでキャンセル
- 初期状態: 有料ユーザー。`/plans` から「サブスクリプション管理」へ移動可能。
- 行動: Customer Portalでキャンセル -> アプリへ戻る。
- 期待値:
  - しばらくしてFreeが「現在のプラン」になる。
  - 画面上でキャンセル反映が確認できる。

## S10: Customer Portalでプラン変更
- 初期状態: Creatorユーザー。
- 行動: Customer PortalでBusinessに変更 -> アプリへ戻る。
- 期待値:
  - Businessが「現在のプラン」になる。
  - 変更結果がUIに反映される。

## S11: メール変更後の課金操作
- 初期状態: 有料ユーザー。メールアドレスを変更済み。
- 行動: `/plans`でプラン変更 or Customer Portalへ移動。
- 期待値:
  - 認証エラーにならずに操作が進む。
  - 変更後のプランがUIに反映される。

## S12: 決済遅延・Webhook遅延
- 初期状態: Checkout完了直後。
- 行動: Webhookが遅延している状況を想定。
- 期待値:
  - 「サブスクリプションの確認中」トーストが表示される。
  - タイムアウト時に再読み込み案内が表示される。
  - 再読み込みで正しいプラン表示になる。
